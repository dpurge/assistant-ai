# Iterative revision

`research-core` is expected to revise reports based on user feedback.

## Common feedback patterns

- **"Add more on X"** — pull additional sources on X, expand that section, keep the rest intact.
- **"Drop Y"** — remove section Y and any citations only it used. Verify other sections didn't lean on those citations.
- **"Verify Z"** — re-check Z's citations against fresh searches. If a citation no longer supports the claim, mark the claim as draft and search for a replacement.
- **"Rewrite for audience A"** — change tone/depth but keep all citations.
- **"Compare with B"** — add a section contrasting the current topic with B; cite both sides.

## Tracking state across revisions

Keep a working note of:

- **Claims accepted** by the user — don't keep re-litigating them.
- **Claims rejected** with reasons — don't reintroduce without new evidence.
- **Citations used** (URL or identifier + retrieval date) — when re-using, re-verify the URL still resolves and the content still says what you cited it for.
- **Open feedback** — explicit asks not yet addressed.

## When to push back

If user feedback would degrade the report (e.g. asking to drop a well-sourced caveat), surface the disagreement once with the reasoning. Defer to the user after that.
